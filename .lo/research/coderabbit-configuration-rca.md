# The Configuration Minefield: What Actually Broke CodeRabbit

The previous research document told a clean story: GitHub's comment re-parenting mechanism was silently breaking CodeRabbit's incremental review tracking. Two agents iterating on the same lines, bookkeeping rewritten underneath them, coverage gaps emerging from infrastructure designed for human reviewers.

That story was plausible. It was also wrong — or at least, it was the wrong layer of the problem.

## What actually happened

We spent an entire session trying to get a simple housekeeping PR to merge. CodeRabbit reviewed, found three cosmetic markdown issues, the CR agent fixed them, and then... silence. CR stopped reviewing. We pushed empty commits. We tagged `@coderabbitai`. Nothing.

The root cause was a single setting buried in CodeRabbit's dashboard:

> **Auto Pause After Reviewed Commits: 5**

After reviewing 5 commits on a PR, CodeRabbit silently pauses. No warning. No comment. No indication that it's stopped watching. It just goes quiet. The CR agent keeps pushing fixes into the void. Auto-merge never fires because the approval never comes.

This is what killed PR #6. This is what killed PR #7. Not GitHub's comment API. Not some deep architectural incompatibility. A default setting that nobody told us about.

## The configuration cascade

Fixing that setting was step one. Step two was getting CodeRabbit to actually *approve* PRs — submit a GitHub `APPROVED` review, not just post comments. Branch protection requires "1 approving review from a reviewer with write access." CodeRabbit was posting `COMMENTED` reviews. Every time.

The fix: `request_changes_workflow: true` in `.coderabbit.yaml`. When enabled, CR submits `REQUEST_CHANGES` when it finds issues and `APPROVED` when everything's clean. This setting defaults to `false`. It's not in the UI. It's yaml-only.

We set it in the org-level config (`looselyorganized/coderabbit/.coderabbit.yaml`). It didn't work. Why? Because every repo had a per-repo `.coderabbit.yaml` generated by `lo-github-sync` with just `reviews: enabled: true`. CodeRabbit's config resolution is **full replacement, not merge** — a per-repo file with one field completely shadows the org config. Our org-level `request_changes_workflow: true` was invisible.

The fix chain:

| Problem | Root cause | Fix |
|---------|-----------|-----|
| CR stops reviewing after N commits | `auto_pause_after_reviewed_commits: 5` (default) | Set to `0` |
| CR comments but never approves | `request_changes_workflow: false` (default) | Set to `true` in org config |
| Org config ignored | Per-repo `.coderabbit.yaml` overrides entirely | Delete per-repo config (or add inheritance) |
| PR won't auto-merge | `gh pr merge --auto` never called | Added to `/lo:ship` feature mode |

Four settings. Four defaults that combine into a system that looks like it's working but silently isn't. Each one is documented if you know where to look. None of them surface errors when misconfigured. The system just... stops doing things.

## The frustration

This is the part that doesn't show up in architecture diagrams.

We spent hours debugging what we thought was a deep infrastructure problem. We wrote a research document proposing a new local review loop architecture. We evaluated CLI tools, designed pluggable interfaces, drew diagrams. The actual fix was changing a number from 5 to 0 in a web dashboard.

The adversarial review loop between CodeRabbit and Claude works. It produces better code. The agents converge. But the configuration surface area around it is hostile. Settings are spread across:

- CodeRabbit web dashboard (per-repo, per-org)
- `.coderabbit.yaml` (per-repo, or in a magic `coderabbit` repo for org-level)
- GitHub branch protection rules
- GitHub repo settings (allow auto-merge)
- GitHub App permissions
- Per-PR auto-merge enablement (`gh pr merge --auto`)

A misconfiguration in any one of these creates a failure mode that looks like something else. CR stops reviewing? Could be rate limits, could be auto-pause, could be the app not installed, could be a title keyword filter. CR reviews but PR won't merge? Could be permissions, could be `request_changes_workflow`, could be the per-repo config shadowing the org config, could be auto-merge not enabled on the PR.

There is no single place that shows you: "here is why this PR is not merging." You debug by elimination across six different systems.

## The corrected picture

The previous document's core thesis — that the review loop should move local — may still be right for performance and coupling reasons. But the urgency was manufactured by a misdiagnosis. The GitHub-mediated loop works fine when configured correctly. The problem was never GitHub's comment API. It was:

1. A default auto-pause threshold we didn't know existed
2. A yaml-only setting that controls whether CR submits approvals
3. A config resolution model that silently shadows org settings

## What we configured

**CodeRabbit org config** (`looselyorganized/coderabbit/.coderabbit.yaml`):
```yaml
reviews:
  enabled: true
  request_changes_workflow: true
  auto_review:
    enabled: true
```

**Per-repo config**: Managed by `lo-github-sync` — only the `enabled: true/false` flag based on project status (Build/Open = true, Explore/Closed = false). All other settings inherit from org.

**Dashboard settings** (org-level):
- Auto Pause After Reviewed Commits: `0` (disabled)
- Automatic Review: on
- Incremental Review: on

**GitHub side**:
- Branch protection: require 1 approving review
- Allow auto-merge: enabled per repo
- `/lo:ship` enables auto-merge on every Build/Open PR

## Lessons

1. **Default settings are opinions.** `auto_pause_after_reviewed_commits: 5` is a reasonable default for human developers. It's catastrophic for an automated agent that pushes 6+ commits per PR. Know your tool's defaults, especially when your usage pattern is non-standard.

2. **Silent failures are the worst failures.** CR didn't error when it auto-paused. It didn't comment. It just stopped. A single log line — "Auto-paused after 5 reviewed commits" — would have saved hours.

3. **Config resolution models matter.** "Per-repo overrides org" sounds simple until your per-repo file has one field and silently shadows twenty org-level settings. Inheritance should be the default, not the exception.

4. **Test the full loop, not the parts.** We verified CR could review. We verified auto-merge was enabled. We verified branch protection. Each component worked. The composition didn't. Integration testing for CI/CD configuration is undervalued.

5. **There must be a better way to automate adversarial code reviews.** The value of the review loop is clear. The operational overhead of configuring and maintaining it across GitHub + CodeRabbit + branch protection + auto-merge is not proportional to the value delivered. Any solution that reduces this surface area — whether local CLI loops, self-contained review actions, or simpler tooling — is worth pursuing.
